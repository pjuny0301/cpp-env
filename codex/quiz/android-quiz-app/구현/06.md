# 06 구현 계획

## 요구사항

6.퀴즈를 옆으로 튕기듯 슬라이드시키면 정답표시같은 거 없이 다음 문제로 바로 넘어감.

몇번이상 문제를 슬라이드시키면 그 문제를 "이미 앎" 그룹으로 옮길 수 있는지 선택 가능.

퀴즈목록의 이미 앎 탭을 터치하면 "이미 앎" 문제를 볼 수 있고 거기서는 "이미 앎"만 있는 퀴즈를 돌릴 수 있음.

그곳 퀴즈 옵션에서는 몇번이상 틀리면 "이미 앎" 그룹에서 뺄지 입력가능

거기서 길게 터치하면 "모름"을 선턕할 수 있는 옵션이 뜸

모름을 선택하면 옵션에서 설정한 숫자와는 상관없이 "이미 앎" 그룹에서 빠지고 본 그룹으로 감

뺴오기 전까지는 앞으로 본문 문제에 더이상 안보임

## 구현 방향

스와이프/이미 앎/모름은 domain FSM으로 구현한다. `skip_question_without_feedback`, `offer_mark_known`, `mark_known_by_swipe`, `mark_unknown`, `known_mode`, `wrong_release` action을 분리하고, UI는 action만 발생시킨다.

## 핵심 산출물

- swipe, known, unknown, wrong release domain actions

## 구현 순서

1. 핵심 산출물인 `swipe, known, unknown, wrong release domain actions`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. swipe next action은 정답 판정과 분리한다.
3. `known` 이동은 swipe count와 옵션값을 session state에서 계산한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`swipe, known, unknown, wrong release domain actions`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
