# 09 구현 계획

## 요구사항

9\. multiselect 유형은 정답을 여러번 선택해야하는 유형, 빈칸에 차례대로 표시되게 함. 속도감있게 하기 위해서 선지에 눌리는 애니메이션은 뺌

## 구현 방향

multiselect는 선택 순서대로 상단 슬롯을 채우는 `multiselect_session_state`를 둔다. 선지 눌림 scale animation은 theme가 별도로 켜지 않는 한 기본 off로 두고, 제출 시 domain의 correct option set과 비교한다.

## 핵심 산출물

- multiselect slot/session state

## 구현 순서

1. 핵심 산출물인 `multiselect slot/session state`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. multiselect 선택 결과를 ordered slot state로 저장한다.
3. 선지 press animation은 token 기본값에서 비활성화해 빠른 입력감을 우선한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`multiselect slot/session state`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
