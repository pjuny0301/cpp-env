# 31 구현 계획

## 요구사항

31\. 빈칸유형에 입력하기 위해서 입력창 터치하면 키보드가 위로 올라오면서 문제가 위로 밀리고 화면밖으로 나가서 스크롤해야 보이는 문제 수정(키보드가 올라올시 레이아웃 변경)

## 구현 방향

키보드가 올라올 때 문제 영역이 화면 밖으로 밀리는 문제는 layout placer에 `keyboard_inset`과 focused input anchoring을 넣어 해결한다. 입력창 포커스 시 question stage는 축소/스크롤 우선순위를 계산하고, 답변 dock은 안전 영역 위에 고정한다.

## 핵심 산출물

- keyboard inset, focused input anchoring

## 구현 순서

1. 핵심 산출물인 `keyboard inset, focused input anchoring`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. keyboard inset과 focused input anchor를 layout state에 넣는다.
3. 문제 본문은 키보드가 올라와도 viewport 밖으로 밀리지 않게 재배치한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`keyboard inset, focused input anchoring`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
