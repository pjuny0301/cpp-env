# 01 구현 계획

## 요구사항

1\. 고성능 AI를 활용해 pdf에 있는 문제만 뽑아다가 챕터별 퀴즈형식으로 만드는 자동화프로그램

## 구현 방향

PDF에서 문제만 추출하는 `pdf-ingest` 워커를 만든다. 단계는 PDF 페이지 추출, 챕터 감지, 문제 후보 분리, AI 검증, `deck/day/question` JSON 컴파일, 에디터 검수 큐 순서다. 모든 프롬프트 입력과 결과는 PDF hash, page range, prompt version으로 캐시한다.

## 핵심 산출물

- pdf-ingest worker and quiz JSON compile

## 구현 순서

1. 핵심 산출물인 `pdf-ingest worker and quiz JSON compile`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. PDF page 추출, 챕터 감지, 문제 후보 분리를 AI 호출 전 deterministic 단계로 나눈다.
3. AI는 문제 후보의 분류/정제만 맡기고 `deck/day/question` JSON compiler가 최종 구조를 만든다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`pdf-ingest worker and quiz JSON compile`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 샘플 PDF 입력, 캐시 재사용, 생성 JSON schema validation으로 확인 가능해야 한다.
