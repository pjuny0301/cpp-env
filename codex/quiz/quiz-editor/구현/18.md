# 18 구현 계획

## 요구사항

18\. 수학 수식 필기 인식 (OCR): 11번(수학 증명) 기능과 연동하여, 태블릿에서 직접 쓴 수식을 텍스트로 변환해 AI에게 전달합니다.

## 구현 방향

17번과 중복이다. 별도 구현하지 않고 같은 `math_ocr` 모듈의 acceptance criteria에 "11번 proof grader와 연동"을 넣는다.

## 핵심 산출물

- math_ocr request/result with stroke or image input

## 구현 순서

1. 핵심 산출물인 `math_ocr request/result with stroke or image input`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. stroke input과 image input을 같은 `math_ocr` request로 보낼 수 있게 한다.
3. 결과는 LaTeX, confidence, bounding box, 원본 input ref를 포함한다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`math_ocr request/result with stroke or image input`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 고정 샘플 입력과 expected grader/OCR output fixture로 확인 가능해야 한다.
