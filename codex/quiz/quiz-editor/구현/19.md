# 19 구현 계획

## 요구사항

19\. **포커스 모드 (Zettelkasten 연동):**  퀴즈를 풀다 막히면 해당 개념과 연결된 자신의 과거 메모(옵시디언 등)를 작은 팝업으로 즉시 띄워주는 기능입니다.(AI가 자신의 손필기내용을 바탕으로 생성해서 띄워줌)

## 구현 방향

포커스 모드는 Obsidian/Zettelkasten vault indexer가 필요하다. 로컬 markdown을 chunk, concept, embedding으로 인덱싱하고, 퀴즈에서 막힌 concept id를 기준으로 작은 popup scene을 띄운다. 손필기 기반 생성물도 같은 artifact ref로 캐시한다.

## 핵심 산출물

- Obsidian/Zettelkasten concept index

## 구현 순서

1. 핵심 산출물인 `Obsidian/Zettelkasten concept index`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. Obsidian/Zettelkasten 노트를 concept id와 embedding index로 연결한다.
3. 팝업 추천은 AI가 생성하되 원본 노트 ref를 함께 노출한다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`Obsidian/Zettelkasten concept index`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
