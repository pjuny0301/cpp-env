# 21 구현 계획

## 요구사항

21\. 개념 의존성 그래프 트래킹: AI가 PDF 지문을 추출할 때(12번), 단순히 텍스트만 나누는 것이 아니라 개념 간의 선후행 관계를 위상학적 트리나 방향성 비순환 그래프(DAG) 형태로 매핑합니다. 만약 14번 기능에서 AI와 토론하다가 학생이 기초 개념을 헷갈려 한다는 것이 파악되면, 즉시 그래프를 역추적하여 해당 선행 개념의 본문이나 덱을 팝업으로 추천해 줍니다.

## 구현 방향

개념 의존성 그래프는 `concept_graph.json` DAG로 저장한다. PDF 인덱싱 시 concept node와 prerequisite edge를 만들고, AI 토론 중 기초 개념 혼동이 감지되면 ancestor node를 역추적해 본문/덱을 추천한다.

## 핵심 산출물

- concept_graph DAG

## 구현 순서

1. 핵심 산출물인 `concept_graph DAG`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. PDF ingest 결과에서 concept_graph DAG를 생성한다.
3. AI 토론 중 약한 선행 개념이 감지되면 graph 역추적으로 추천 deck/content를 찾는다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`concept_graph DAG`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 샘플 PDF 입력, 캐시 재사용, 생성 JSON schema validation으로 확인 가능해야 한다.
