# 22 구현 계획

## 요구사항

22\. 코드/로직 스니펫 샌드박스 채점: 수학 증명(11번)과 결합하여, 프로그래밍이나 알고리즘 문제의 경우 학생이 직접 코드를 작성하면 샌드박스 환경에서 컴파일 및 실행 결과를 검증하는 유형입니다. 단순히 결과만 보는 것이 아니라, AI가 작성된 코드의 템플릿 구조나 메모리 라이프사이클 등을 분석해 피드백을 주도록 할 수 있습니다.

## 구현 방향

코드/로직 샌드박스는 proof grader와 별도 워커로 둔다. compile/run은 제한된 컨테이너나 Windows sandbox에서 실행하고, AI는 실행 결과, 테스트 실패, 코드 구조 요약만 받아 피드백한다.

## 핵심 산출물

- compile/run code sandbox worker

## 구현 순서

1. 핵심 산출물인 `compile/run code sandbox worker`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. 코드 실행은 격리된 sandbox worker가 compile/run result만 반환한다.
3. AI 피드백은 실행 로그와 코드 구조 분석을 별도 단계로 생성한다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`compile/run code sandbox worker`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 고정 샘플 입력과 expected grader/OCR output fixture로 확인 가능해야 한다.
