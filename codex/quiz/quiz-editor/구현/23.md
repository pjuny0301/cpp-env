# 23 구현 계획

## 요구사항

23\. ​리버스 엔지니어링 / 디버깅 퀴즈: AI가 의도적으로 논리적 비약이 있는 수학 증명이나, 미세한 버그(예: 참조 순환 등)가 숨겨진 개념 설명을 제시합니다. 학생이 그 '오류'를 찾아내어 지적하게 만드는 역할극 형태의 퀴즈로, 메타 인지를 강하게 자극할 수 있습니다.

## 구현 방향

리버스 엔지니어링/디버깅 퀴즈는 `question_type=debugging` 또는 `scenario_kind=reverse_debug`로 추가한다. AI가 일부러 넣은 오류는 metadata에 숨겨 저장하고, 학생 답변은 오류 위치/이유/수정안을 기준으로 채점한다.

## 핵심 산출물

- debugging/reverse_debug question type

## 구현 순서

1. 핵심 산출물인 `debugging/reverse_debug question type`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. reverse_debug question type을 추가하고 오류가 숨어 있는 설명/증명을 seed로 저장한다.
3. 학생의 지적은 오류 위치, 이유, 수정안을 따로 채점한다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`debugging/reverse_debug question type`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 고정 샘플 입력과 expected grader/OCR output fixture로 확인 가능해야 한다.
