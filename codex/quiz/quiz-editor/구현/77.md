# 77 구현 계획

## 요구사항

77\. Deck에 pdf 넣으면 서버로 전송 -> 텍스트화, 퀴즈화후 다시 폰으로 전송 -> 다음 앱에 접속시 \[...Deck pdf분석완료되었습니다. 다운로드받으시겠습니까?] -> 아니요 누르면 Deck 접속하기 전에 서버에서 분석된 pdf 다운로드받기 누를 수 있음. -> 업로드하는 거, 분석하는 거는 각각 고전 카툰 애니메이션으로 표현

## 구현 방향

Deck에 PDF 업로드하면 서버 job으로 보내고, 완료 후 앱 접속 시 다운로드 prompt를 띄운다. upload/progress/analyze/done 상태는 job id로 poll 또는 push하며, 업로드/분석 animation은 theme asset으로 교체 가능하게 한다.

## 핵심 산출물

- server PDF analysis job and download prompt

## 구현 순서

1. 핵심 산출물인 `server PDF analysis job and download prompt`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. PDF 업로드 job, 분석 job, 다운로드 prompt를 상태별로 나눈다.
3. 업로드/분석 진행 표현은 고전 카툰 animation asset으로 연결한다.
4. worker/service command와 result artifact 형식을 정한다.
5. 에디터에는 적용 전 diff 또는 review queue로 연결한다.
6. cache miss, retry, partial failure 상태를 manifest에 남긴다.

## 대안

`server PDF analysis job and download prompt`를 외부 워커/캐시 artifact로 분리하지 않고 앱이나 에디터 내부에서 AI를 즉시 호출해 결과를 바로 반영하는 방식이 있다.

## 장단점

장점:
- 무거운 AI 작업을 앱 런타임에서 분리해 모바일/렌더러 성능을 보호한다.
- 입력, prompt version, 결과 artifact가 남아 같은 작업을 재호출하지 않고 검수할 수 있다.
- 에디터와 Vulkan/Android 앱이 같은 생성 결과를 재사용할 수 있다.

단점:
- 초기에 worker command, cache key, artifact schema를 맞추는 비용이 든다.
- 즉시 반영형 UX보다 job 상태와 review queue를 더 많이 설계해야 한다.

## 구현 기준

- 같은 입력은 같은 cache key로 재사용되어야 한다.
- AI 결과는 사람이 검수 가능한 patch나 artifact로 남아야 한다.
- 앱 런타임은 무거운 AI/OCR/분석 호출을 직접 수행하지 않아야 한다.
- 샘플 PDF 입력, 캐시 재사용, 생성 JSON schema validation으로 확인 가능해야 한다.
