# 05 구현 계획

## 요구사항

5\. 위 프로젝트를 만드는데 필요한 요구사항 트리로 표현(블로그에 있는 기법과 관련시킴)

## 구현 방향

별도 트리 파일은 유지하지 않는다. 요구사항 트리 역할은 `codex/quiz/big_plan.md`, 루트 `codex/quiz/구현/NN.md`, `quiz-vulkan/big_plan.md`, `quiz-vulkan/구현/NN.md`의 폭포수 문서 흐름으로 흡수한다.

## 핵심 산출물

- 폭포수 문서 연결 구조: quiz big plan, root handoff, Vulkan big plan, Vulkan implementation plan

## 구현 순서

1. `big_plan.md`에는 요구사항 원문과 폭포수 단계만 둔다.
2. 루트 `구현/NN.md`에는 담당 프로젝트와 하위 문서 진입점만 둔다.
3. `quiz-vulkan/big_plan.md`에는 Vulkan 내부 단계, 선행조건, 구현 문서 인덱스를 둔다.
4. `quiz-vulkan/구현/NN.md`에는 말로 된 세부 구현계획, 대안, 장단점, 구현 기준을 둔다.
5. 새 요구가 들어오면 이 네 단계 문서가 끊기지 않도록 링크와 담당 범위를 함께 갱신한다.

## 대안

별도 트리 파일을 되살려 wave/owner/status 압축표를 따로 유지하는 방식이 있다.

## 장단점

장점:
- 문서 흐름이 `big_plan -> quiz/구현 -> project big_plan -> project/구현`으로 단순해진다.
- 별도 tree 문서와 구현 문서가 서로 어긋나는 문제를 줄인다.
- 구현자는 번호별 하위 문서만 따라가면 된다.

단점:
- wave/owner/status를 한 화면에 압축해서 보는 별도 표는 사라진다.
- 하위 문서 링크를 정확히 유지하지 않으면 흐름을 따라가기 어려워진다.

## 구현 기준

- 루트와 프로젝트 문서 사이에 중복된 세부 구현계획이 없어야 한다.
- `quiz-vulkan/big_plan.md`의 인덱스가 실제 `quiz-vulkan/구현/NN.md` 파일로 연결되어야 한다.
- 요구사항별 세부 구현은 코드가 아니라 prose plan으로 작성되어야 한다.
