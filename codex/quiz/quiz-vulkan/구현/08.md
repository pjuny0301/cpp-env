# 08 구현 계획

## 요구사항

8\. select 유형에서 OX표시 제거. multiselect multiblank blank 유형에선 OX 표시 이전으로 되돌림

## 구현 방향

select 유형의 큰 OX 오버레이는 금지한다. blank, multi_blank, multiselect에는 유형별로 낮은 불투명도의 inline feedback을 허용하되, 화면 중앙 오버레이는 쓰지 않는다.

## 핵심 산출물

- 큰 OX overlay 금지, inline feedback 정책

## 구현 순서

1. 핵심 산출물인 `큰 OX overlay 금지, inline feedback 정책`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. select 유형과 multiselect/multiblank/blank 유형의 feedback policy를 분리한다.
3. 큰 OX overlay는 select에서 금지하고 나머지 유형은 기존 inline feedback로 복구한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`큰 OX overlay 금지, inline feedback 정책`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
