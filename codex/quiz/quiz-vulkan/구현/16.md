# 16 구현 계획

## 요구사항

16\. 기능을 모듈화(프로그램으로 나누고, 각 기능이 공유하는 일반화된 표준 개발

## 구현 방향

기능은 독립 프로그램과 표준 계약으로 나눈다. 최소 프로그램은 player, editor, pdf-indexer, ai-generator, translation-cache, zettel-indexer, proof-grader, code-sandbox, sync-api다. 공통 schema version을 두고 breaking change는 migration으로 처리한다.

## 핵심 산출물

- 프로그램 분리 계약: player, editor, pdf-indexer, ai-generator, cache, grader

## 구현 순서

1. 핵심 산출물인 `프로그램 분리 계약: player, editor, pdf-indexer, ai-generator, cache, grader`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. player, editor, pdf-indexer, ai-generator, cache, grader의 입출력 계약을 분리한다.
3. 공유 표준은 schema와 command/result envelope부터 고정한다.
4. 관련 owner boundary와 data contract를 먼저 문서화한다.
5. 하위 프로젝트가 참조할 최소 schema, FSM, cache, test 기준을 정한다.
6. 새 요구가 들어오면 같은 번호 구현 문서와 상위 인덱스를 함께 갱신한다.

## 대안

`프로그램 분리 계약: player, editor, pdf-indexer, ai-generator, cache, grader`를 먼저 계약화하지 않고 기능별로 필요한 부분만 즉흥적으로 구현하는 방식이 있다.

## 장단점

장점:
- owner boundary와 데이터 계약이 먼저 잡혀 하위 프로젝트 간 충돌을 줄인다.
- 큰 계획을 작은 산출물과 테스트 기준으로 나누어 후속 구현자가 바로 이어갈 수 있다.
- 토큰을 아끼면서도 필요한 문서를 번호별로 찾아 읽을 수 있다.

단점:
- 실제 기능 구현 전에 문서와 계약을 정리하는 시간이 든다.
- 계약이 바뀌면 관련 번호 문서와 인덱스를 함께 갱신해야 한다.

## 구현 기준

- 구현자가 owner boundary, 계약, 작업 순서를 문서만 보고 판단할 수 있어야 한다.
- 공통 계약 변경은 하위 앱별 임시 구현보다 먼저 고정되어야 한다.
- 큰 자료는 인덱스와 요약을 먼저 읽는 구조여야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
