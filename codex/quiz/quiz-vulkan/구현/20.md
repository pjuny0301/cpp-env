# 20 구현 계획

## 요구사항

20\. **망각 곡선 기반 에이징 (Spaced Repetition System, SRS):**  6번의 "이미 앎" 그룹에 들어간 문제에 시간 가중치를 부여하는 기능입니다. 며칠이 지나면 망각 곡선에 따라 해당 문제를 "이미 앎"에서 다시 "본 그룹"으로 자연스럽게 복귀시켜 장기 기억으로 전환되었는지 점검합니다.

## 구현 방향

SRS aging은 `known` 상태에 `known_since`, `due_at`, `ease`, `review_interval`을 추가한다. due가 지나면 본 그룹으로 자연 복귀시키거나 "복습 예정" 탭에 노출하고, 정답/오답 결과로 interval을 갱신한다.

## 핵심 산출물

- SRS known aging and due_at policy

## 구현 순서

1. 핵심 산출물인 `SRS known aging and due_at policy`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. known bucket에 `known_at`, `due_at`, 실패 횟수, 복귀 정책을 저장한다.
3. 망각 곡선 계산은 UI가 아니라 domain scheduler가 담당한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`SRS known aging and due_at policy`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
