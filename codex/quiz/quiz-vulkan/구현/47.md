# 47 구현 계획

## 요구사항

47\. 단답형에 OX 표시 추가할 땐 화면과 같은 색계열에서 흐리게 투명도 높게 한다
또한 답은 퀴즈 바로 아래에 짧은 텍스트일 경우 굵은 글씨로,
긴 텍스트인 경우 표시하는 곳에 따로
정답은 문제를 위로 밀면서 그곳에 표시
기존 빨간글씨로 정답 표기하는 것은 제거

## 구현 방향

단답형 OX는 화면과 같은 색 계열의 높은 투명도 feedback으로 제한한다. 짧은 정답은 문제 바로 아래 bold inline, 긴 정답은 별도 answer panel에 표시하며, 정답 표시가 문제를 위로 자연스럽게 밀도록 layout rule을 둔다. 기존 빨간 글씨 정답 표기는 제거한다.

## 핵심 산출물

- 단답형 inline feedback 정책

## 구현 순서

1. 핵심 산출물인 `단답형 inline feedback 정책`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. 단답형 inline feedback 색은 현재 화면 색계열에서 alpha가 높은 token으로 계산한다.
3. 큰 overlay나 강한 빨간색은 사용하지 않는다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`단답형 inline feedback 정책`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
