# 52 구현 계획

## 요구사항

52\. 정답 맞았을 때 나머지 흐려지는 거 없앤다

## 구현 방향

다른 브랜치 작업으로 분리한다. 정답 시 나머지 선지를 흐리게 하는 처리는 style policy에서 off로 둔다.

## 핵심 산출물

- dim rest options policy

## 구현 순서

1. 핵심 산출물인 `dim rest options policy`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. dim rest options policy를 token으로 분리한다.
3. 나머지 선지 흐림 제거는 renderer가 아니라 scene builder 정책으로 처리한다.
4. 현재 줄기에서는 token 이름, 기본값, fallback만 고정한다.
5. 실제 화면/애니메이션 변경은 분리 브랜치에서 scene builder 정책으로 구현한다.
6. merge 시 기존 theme과 deck fixture가 그대로 동작하는지 확인한다.

## 대안

`dim rest options policy`를 분리 브랜치로 넘기지 않고 현재 줄기에서 바로 UI/animation까지 구현하는 방식이 있다.

## 장단점

장점:
- 현재 줄기에는 token 이름과 fallback만 남겨 merge 충돌을 줄인다.
- 다른 브랜치의 시각 실험이 기존 domain과 deck fixture를 깨뜨리지 않는다.

단점:
- 실제 화면 변화는 해당 브랜치가 합쳐질 때까지 확인하기 어렵다.
- token 정책과 실제 구현이 분리되어 문서 동기화가 필요하다.

## 구현 기준

- 현재 줄기에는 큰 UI 변경을 섞지 않고 정책 이름과 fallback만 남겨야 한다.
- 분리 브랜치 구현은 token 기본값으로 켜고 끌 수 있어야 한다.
- merge 후 기존 deck과 theme이 그대로 렌더링되어야 한다.
- scene snapshot, layout test, renderer smoke test로 확인 가능해야 한다.
