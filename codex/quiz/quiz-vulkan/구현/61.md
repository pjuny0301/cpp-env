# 61 구현 계획

## 요구사항

61\. 다중빈칸카드유형 주가, 위쪽에 카드를 많이 띄워놓고 전부 배치하면 어느 부분이 오답인지 표기

## 구현 방향

다중빈칸 카드 유형은 `multi_blank_card` presentation으로 추가한다. 위쪽 카드 tray에 선택지를 많이 띄우고, 배치 완료 후 slot별 correct/incorrect를 표시한다.

## 핵심 산출물

- multi_blank_card presentation

## 구현 순서

1. 핵심 산출물인 `multi_blank_card presentation`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. multi_blank_card presentation을 별도 question presentation으로 둔다.
3. 배치 후 오답 위치 표시는 slot별 상태와 anchor로 처리한다.
4. domain action과 session state 이름을 먼저 정한다.
5. Android 기준 동작을 고정한 뒤 Vulkan/domain 테스트로 옮길 수 있게 한다.
6. UI는 action 발생과 state 표시만 맡고 정책 판단은 domain에 둔다.

## 대안

`multi_blank_card presentation`를 domain/session state로 정의하지 않고 각 UI 컴포넌트에서 직접 상태를 관리하는 방식이 있다.

## 장단점

장점:
- Android 기준 UX와 Vulkan 포팅이 같은 action/state 의미를 공유한다.
- 제스처, 옵션, navigation, 저장 상태를 UI 없이 테스트할 수 있다.
- 렌더러나 화면 구현을 바꿔도 학습 규칙이 흔들리지 않는다.

단점:
- 작은 UX 변경도 domain action과 fallback을 먼저 정해야 하므로 초기 작업량이 늘어난다.
- 기존 UI 임시 상태를 session state로 옮기는 정리 작업이 필요하다.

## 구현 기준

- 상태 변화는 UI 임시 상태가 아니라 domain/session state로 남아야 한다.
- Android 기준 UX와 Vulkan domain이 같은 action 의미를 공유해야 한다.
- 새 옵션이 꺼진 상태에서는 기존 풀이 흐름이 유지되어야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
