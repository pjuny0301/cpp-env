# 71 구현 계획

## 요구사항

71\. 데톱앱으로 학습시 코덱스봇 캐릭터 왼쪽에 코덱스봇과 세로로 비슷한 크기의 말풍선에 설명 씌여있는 형식도 있음 코덱스봇+말풍선의 크기는, 가로로는 화면이 꽉차고 세로로는 1/3 \~ 1/5 정도

## 구현 방향

데스크톱 학습의 Codexbot+말풍선은 `study_companion_scene`으로 만든다. 화면 가로를 크게 쓰되 세로 1/3-1/5 범위 layout constraint를 두고, 대화 텍스트는 content script artifact에서 읽는다.

## 핵심 산출물

- study companion scene

## 구현 순서

1. 핵심 산출물인 `study companion scene`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. study companion scene을 Codexbot과 말풍선 layout으로 정의한다.
3. 화면 가로를 채우고 세로 1/3~1/5 범위에 들어오게 constraint를 둔다.
4. domain state를 scene builder 입력으로 넘기고 renderer에는 draw command만 전달한다.
5. style, layout, motion은 theme token과 fallback token으로 분리한다.
6. desktop과 Android surface가 같은 scene/layout 계약을 쓰게 한다.

## 대안

`study companion scene`를 scene/layout/theme token으로 분리하지 않고 renderer나 화면 코드에 직접 하드코딩하는 방식이 있다.

## 장단점

장점:
- 테마, 카툰 스타일, desktop/Android surface가 같은 scene 계약을 재사용한다.
- renderer는 draw command만 처리하므로 문제 유형별 규칙과 그래픽 코드가 섞이지 않는다.
- 디자인 변경을 token 교체나 scene builder 정책 변경으로 좁힐 수 있다.

단점:
- scene schema, token fallback, snapshot 기준을 먼저 만들어야 해서 첫 화면 구현이 느리다.
- 단순한 시각 변경도 theme/scene 계층을 통과해야 하므로 파일 수가 늘 수 있다.

## 구현 기준

- 시각 정책은 theme token, scene command, layout constraint로 교체 가능해야 한다.
- renderer에는 문제 유형별 비즈니스 규칙을 넣지 않아야 한다.
- token이 일부 빠져도 fallback으로 화면이 깨지지 않아야 한다.
- scene snapshot, layout test, renderer smoke test로 확인 가능해야 한다.
