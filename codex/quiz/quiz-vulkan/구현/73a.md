# 73a 구현 계획

## 요구사항

73\. 말풍선은 마치 게임의 npc대화창처럼 구성되어있고 뒤로 넘기는 식으로 대화함. 코덱스봇과 대화하다보면 연필이 적절한 위치에 뜨면서 부수적인 설명을 하기도 함. 그리고 단어나 문장, 문단에 형광펜 색칠되며 코덱스봇이 그에 대한 설명을 하기도 함.

## 구현 방향

NPC 대화창, 다음 대사 넘김, 형광펜 highlight는 scripted interaction interpreter로 만든다. script는 dialogue lines, speaker, target text range, highlight color, optional pencil note를 포함한다.

## 핵심 산출물

- scripted dialogue, highlight interpreter

## 구현 순서

1. 핵심 산출물인 `scripted dialogue, highlight interpreter`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. scripted dialogue interpreter가 NPC 대화창, highlight, pencil annotation event를 처리한다.
3. 대화 넘김과 형광펜 강조는 같은 timeline event로 관리한다.
4. domain state를 scene builder 입력으로 넘기고 renderer에는 draw command만 전달한다.
5. style, layout, motion은 theme token과 fallback token으로 분리한다.
6. desktop과 Android surface가 같은 scene/layout 계약을 쓰게 한다.

## 대안

`scripted dialogue, highlight interpreter`를 scene/layout/theme token으로 분리하지 않고 renderer나 화면 코드에 직접 하드코딩하는 방식이 있다.

## 장단점

장점:
- 테마, 카툰 스타일, desktop/Android surface가 같은 scene 계약을 재사용한다.
- renderer는 draw command만 처리하므로 문제 유형별 규칙과 그래픽 코드가 섞이지 않는다.
- 디자인 변경을 token 교체나 scene builder 정책 변경으로 좁힐 수 있다.

단점:
- scene schema, token fallback, snapshot 기준을 먼저 만들어야 해서 첫 화면 구현이 느리다.
- 단순한 시각 변경도 theme/scene 계층을 통과해야 하므로 파일 수가 늘 수 있다.

## 구현 기준

- 시각 정책은 theme token, scene command, layout constraint로 교체 가능해야 한다.
- renderer에는 문제 유형별 비즈니스 규칙을 넣지 않아야 한다.
- token이 일부 빠져도 fallback으로 화면이 깨지지 않아야 한다.
- scene snapshot, layout test, renderer smoke test로 확인 가능해야 한다.
