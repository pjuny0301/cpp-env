# 73b 구현 계획

## 요구사항

73\. AI가 배정된 챕터의 텍스트를 받고 71 72를 짜되 앞으로 다시 생성하는 일 없게 파일로 생성결과물은 따로 저장(주로 UI 형식과 같이 다양한 상호작용을 가정한 형식으로 저장하게끔. 그 형식을 해석하는 해석기를 그리 디자인해서 형식을 그렇게 저장하게끔 하면 되겠네)

## 구현 방향

AI가 챕터 텍스트를 받아 71-73a script를 생성하되, 재생성하지 않도록 `chapter_study_script.json`으로 저장한다. renderer는 이 형식을 해석만 하고, AI 호출은 generation worker에서만 수행한다.

## 핵심 산출물

- cached chapter study script generation

## 구현 순서

1. 핵심 산출물인 `cached chapter study script generation`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. AI 생성 결과를 `chapter_study_script.json` 같은 재사용 파일로 저장한다.
3. renderer는 script를 해석만 하고 AI 호출은 generation worker에 둔다.
4. domain state를 scene builder 입력으로 넘기고 renderer에는 draw command만 전달한다.
5. style, layout, motion은 theme token과 fallback token으로 분리한다.
6. desktop과 Android surface가 같은 scene/layout 계약을 쓰게 한다.

## 대안

`cached chapter study script generation`를 scene/layout/theme token으로 분리하지 않고 renderer나 화면 코드에 직접 하드코딩하는 방식이 있다.

## 장단점

장점:
- 테마, 카툰 스타일, desktop/Android surface가 같은 scene 계약을 재사용한다.
- renderer는 draw command만 처리하므로 문제 유형별 규칙과 그래픽 코드가 섞이지 않는다.
- 디자인 변경을 token 교체나 scene builder 정책 변경으로 좁힐 수 있다.

단점:
- scene schema, token fallback, snapshot 기준을 먼저 만들어야 해서 첫 화면 구현이 느리다.
- 단순한 시각 변경도 theme/scene 계층을 통과해야 하므로 파일 수가 늘 수 있다.

## 구현 기준

- 시각 정책은 theme token, scene command, layout constraint로 교체 가능해야 한다.
- renderer에는 문제 유형별 비즈니스 규칙을 넣지 않아야 한다.
- token이 일부 빠져도 fallback으로 화면이 깨지지 않아야 한다.
- scene snapshot, layout test, renderer smoke test로 확인 가능해야 한다.
