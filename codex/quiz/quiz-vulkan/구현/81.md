# 81 구현 계획

## 요구사항

81\. 토큰관리햐기 위해선 어찌해야하는지?

## 구현 방향

토큰 관리는 `codex/common/agent/shared_command.json`, subtree `AGENTS.md`, hash cache, prompt summaries, artifact manifests로 한다. AI 호출 전에는 cache lookup, chunk selection, max token budget, expected output schema를 기록한다.

## 핵심 산출물

- cache key, shared command, prompt summary, artifact manifest 규칙

## 구현 순서

1. 핵심 산출물인 `cache key, shared command, prompt summary, artifact manifest 규칙`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. cache key, shared command, prompt summary, artifact manifest 규칙을 통일한다.
3. 큰 자료는 원문 대신 요약/manifest를 먼저 읽는 흐름을 강제한다.
4. 관련 owner boundary와 data contract를 먼저 문서화한다.
5. 하위 프로젝트가 참조할 최소 schema, FSM, cache, test 기준을 정한다.
6. 새 요구가 들어오면 같은 번호 구현 문서와 상위 인덱스를 함께 갱신한다.

## 대안

`cache key, shared command, prompt summary, artifact manifest 규칙`를 먼저 계약화하지 않고 기능별로 필요한 부분만 즉흥적으로 구현하는 방식이 있다.

## 장단점

장점:
- owner boundary와 데이터 계약이 먼저 잡혀 하위 프로젝트 간 충돌을 줄인다.
- 큰 계획을 작은 산출물과 테스트 기준으로 나누어 후속 구현자가 바로 이어갈 수 있다.
- 토큰을 아끼면서도 필요한 문서를 번호별로 찾아 읽을 수 있다.

단점:
- 실제 기능 구현 전에 문서와 계약을 정리하는 시간이 든다.
- 계약이 바뀌면 관련 번호 문서와 인덱스를 함께 갱신해야 한다.

## 구현 기준

- 구현자가 owner boundary, 계약, 작업 순서를 문서만 보고 판단할 수 있어야 한다.
- 공통 계약 변경은 하위 앱별 임시 구현보다 먼저 고정되어야 한다.
- 큰 자료는 인덱스와 요약을 먼저 읽는 구조여야 한다.
- domain unit test와 주요 UI flow smoke test로 확인 가능해야 한다.
