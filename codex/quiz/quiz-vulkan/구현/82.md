# 82 구현 계획

## 요구사항

82\. vulkan 렌더러로 구현 - 순수 c++

## 구현 방향

Vulkan renderer 구현은 `apps/quiz/quiz-vulkan`에서 순수 C++로 진행한다. 1차는 draw command counter가 아닌 실제 window/swapchain/quad/text/image, 2차는 retained scene UI, 3차는 Android surface, 4차는 cartoon/theme renderer 순서로 간다.

## 핵심 산출물

- Vulkan MVP order: desktop, retained UI, Android surface, cartoon/theme

## 구현 순서

1. 핵심 산출물인 `Vulkan MVP order: desktop, retained UI, Android surface, cartoon/theme`의 입력, 출력, 저장/적용 지점을 먼저 고정한다.
2. desktop window/swapchain/quad/text/image 경로를 먼저 완성한다.
3. retained scene UI, Android surface, cartoon/theme renderer 순서로 확장한다.
4. domain state를 scene builder 입력으로 넘기고 renderer에는 draw command만 전달한다.
5. style, layout, motion은 theme token과 fallback token으로 분리한다.
6. desktop과 Android surface가 같은 scene/layout 계약을 쓰게 한다.

## 대안

`Vulkan MVP order: desktop, retained UI, Android surface, cartoon/theme`를 scene/layout/theme token으로 분리하지 않고 renderer나 화면 코드에 직접 하드코딩하는 방식이 있다.

## 장단점

장점:
- 테마, 카툰 스타일, desktop/Android surface가 같은 scene 계약을 재사용한다.
- renderer는 draw command만 처리하므로 문제 유형별 규칙과 그래픽 코드가 섞이지 않는다.
- 디자인 변경을 token 교체나 scene builder 정책 변경으로 좁힐 수 있다.

단점:
- scene schema, token fallback, snapshot 기준을 먼저 만들어야 해서 첫 화면 구현이 느리다.
- 단순한 시각 변경도 theme/scene 계층을 통과해야 하므로 파일 수가 늘 수 있다.

## 구현 기준

- 시각 정책은 theme token, scene command, layout constraint로 교체 가능해야 한다.
- renderer에는 문제 유형별 비즈니스 규칙을 넣지 않아야 한다.
- token이 일부 빠져도 fallback으로 화면이 깨지지 않아야 한다.
- scene snapshot, layout test, renderer smoke test로 확인 가능해야 한다.
