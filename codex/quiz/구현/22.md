# 22 상위 구현 연결

## 요구사항

22\. 코드/로직 스니펫 샌드박스 채점: 수학 증명(11번)과 결합하여, 프로그래밍이나 알고리즘 문제의 경우 학생이 직접 코드를 작성하면 샌드박스 환경에서 컴파일 및 실행 결과를 검증하는 유형입니다. 단순히 결과만 보는 것이 아니라, AI가 작성된 코드의 템플릿 구조나 메모리 라이프사이클 등을 분석해 피드백을 주도록 할 수 있습니다.

## 상위 이행 방향

이 루트 문서는 요구사항 22를 세부 구현하지 않고 `quiz-editor`의 waterfall 작업으로 내려보내는 상위 연결 문서다. 실제 구현 방식, 코드 경계, 검증 기준은 하위 프로젝트 `big_plan.md`와 `구현/22.md`에서 관리한다.

## 담당 프로젝트

- `quiz-editor`: [quiz-editor/big_plan.md](../quiz-editor/big_plan.md), [quiz-editor/구현/22.md](../quiz-editor/구현/22.md) - AI/데이터 제작, 검수, Git 저장, worker artifact 연결을 담당한다.

## 폭포수 이행 순서

1. 루트 `big_plan.md`에서 요구사항 원문을 확인한다.
2. 이 문서에서 담당 프로젝트와 하위 문서 진입점을 확인한다.
3. `quiz-editor/big_plan.md`에서 프로젝트 내 단계와 선행조건을 확인한다.
4. `quiz-editor/구현/22.md`에서 실제 구현 방향, 대안, 장단점, 구현 기준을 따른다.

## 대안

루트 `quiz/구현` 문서에 모든 세부 구현계획을 쓰고, 하위 프로젝트 `big_plan.md`와 `구현/`에는 같은 내용을 복사하는 방식이 있다.

## 장단점

장점:
- 상위 문서는 요구사항과 하위 진입점만 관리하므로 프로젝트 세부 구현과 겹치지 않는다.
- 구현자는 `big_plan -> quiz/구현 -> project big_plan -> project/구현` 순서로 읽으면 된다.
- 프로젝트별 세부 구현 변경이 루트 문서와 충돌할 가능성이 줄어든다.

단점:
- 한 요구사항을 완전히 이해하려면 루트 문서와 담당 프로젝트 문서를 함께 열어야 한다.
- 담당 프로젝트가 바뀌면 루트 연결 문서와 프로젝트 인덱스를 같이 갱신해야 한다.

## 이행 기준

- 루트 문서에는 하위 프로젝트 세부 구현 순서나 코드 경계를 중복 작성하지 않는다.
- 모든 담당 프로젝트의 `big_plan.md`와 `구현/NN.md` 링크가 실제 파일로 열려야 한다.
- 하위 구현이 바뀌면 해당 프로젝트 문서를 먼저 고치고, 루트 문서는 담당/흐름 변경이 있을 때만 갱신한다.
